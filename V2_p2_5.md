# 2.5. Конфигурация инструментов разработки

Эта инструкция описывает настройку инструментов разработки для обеспечения единого стиля кода, высокого качества и автоматизации проверок в проекте.

## 1. Настройка стиля кода

Цель — обеспечить одинаковое форматирование кода у всех участников команды, независимо от их локальных настроек IDE.

### 1.1. EditorConfig
-   **Назначение:** Файл `.editorconfig` в корне проекта позволяет определить базовые правила стиля кода (размер отступа, кодировка, конечные пробелы и т.д.).
-   **Конфигурация:**
    -   **Android Studio:** Поддержка `.editorconfig` включена по умолчанию. Никаких дополнительных действий не требуется.
    -   **Visual Studio Code:** Убедитесь, что установлено расширение `EditorConfig for VS Code`.
-   **Действие:** Откройте файл `.editorconfig` и убедитесь, что его настройки соответствуют требованиям проекта. Стандартные настройки для Kotlin/Android уже должны быть в нем.

### 1.2. Стиль кода Kotlin в Android Studio
-   **Назначение:** Применение официального стиля кода от JetBrains для Kotlin.
-   **Конфигурация:**
    1.  Откройте настройки Android Studio: **File > Settings...**
    2.  Перейдите в **Editor > Code Style > Kotlin**.
    3.  В правом верхнем углу нажмите на **"Set from..."**.
    4.  Выберите **"Kotlin style guide"**.
    5.  Нажмите **Apply** и **OK**.
-   **Применение:** Для форматирования файла в соответствии с настроенным стилем используйте комбинацию клавиш `Ctrl+Alt+L`.

## 2. Статический анализ кода (Android Lint)

-   **Назначение:** Android Lint — это инструмент статического анализа, который проверяет исходный код на наличие потенциальных ошибок, проблем с производительностью, безопасностью, доступностью и стилем.
-   **Конфигурация:**
    1.  Настройка правил Lint производится в файле `app/build.gradle.kts` внутри блока `android { ... }`.
    2.  Пример конфигурации для включения всех проверок, но отключения конкретного правила и прекращения сборки при наличии ошибок:
        ```kotlin
        android {
            // ...
            lint {
                checkAllWarnings = true
                abortOnError = true
                disable.add("SpecificWarningId") // Пример отключения правила
            }
        }
        ```
-   **Baseline (Базовый уровень):**
    -   Файл `app/lint-baseline.xml` в проекте содержит список уже существующих проблем. Lint использует его, чтобы сообщать только о *новых* проблемах, появившихся после создания baseline. Это позволяет постепенно улучшать качество кода, не исправляя все старые ошибки сразу.
    -   Для обновления baseline выполните команду: `./gradlew lint -Pandroid.lint.baselines.continue=true`
-   **Запуск:**
    -   Для запуска анализа выполните в терминале:
        ```bash
        ./gradlew lintDebug
        ```
    -   Отчет будет сгенерирован в `app/build/reports/lint-results-debug.html`.

## 3. Конфигурация сборки

### 3.1. Build Variants (Варианты сборки)
-   **Назначение:** Android Studio позволяет работать с двумя основными вариантами сборки: `debug` и `release`.
-   **Переключение:** Выбрать активный вариант можно в окне **Build Variants** (обычно находится на левой боковой панели).
-   **Конфигурация:** В `app/build.gradle.kts` можно задать разные параметры для каждого типа сборки. Например, добавить суффикс к ID приложения для debug-сборок:
    ```kotlin
    android {
        // ...
        buildTypes {
            getByName("debug") {
                applicationIdSuffix = ".debug"
            }
            getByName("release") {
                // ...
            }
        }
    }
    ```

### 3.2. Файл `local.properties`
-   **Назначение:** Этот файл предназначен для хранения локальных настроек разработчика, которые не должны попадать в систему контроля версий (например, путь к Android SDK `sdk.dir`).
-   **Конфигурация:** Убедитесь, что `local.properties` добавлен в `.gitignore`. Android Studio обычно управляет этим файлом автоматически.

## 4. (Опционально) Git Hooks для локальных проверок

-   **Назначение:** Автоматический запуск скриптов перед созданием коммита для предотвращения попадания в репозиторий кода с ошибками.
-   **Настройка (Пример для `pre-commit`):
    1.  Создайте файл `pre-commit` в директории `.git/hooks/`:
        ```bash
        touch .git/hooks/pre-commit
        ```
    2.  Сделайте его исполняемым:
        ```bash
        chmod +x .git/hooks/pre-commit
        ```
    3.  Добавьте в него следующий скрипт, который запускает тесты и lint перед коммитом:
        ```sh
        #!/bin/sh
        echo "Running pre-commit hook: running tests and lint..."

        # Запуск тестов
        ./gradlew testDebugUnitTest
        if [ $? -ne 0 ]; then
            echo "Unit tests failed. Commit aborted."
            exit 1
        fi

        # Запуск lint
        ./gradlew lintDebug
        if [ $? -ne 0 ]; then
            echo "Lint checks failed. Commit aborted."
            exit 1
        fi

        echo "All checks passed. Proceeding with commit."
        exit 0
        ```
-   **Действие:** Теперь при каждой попытке выполнить `git commit` сначала будут запущены unit-тесты и lint-проверки. Если любая из задач завершится с ошибкой, коммит будет отменен.
