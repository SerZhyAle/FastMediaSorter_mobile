2.1. Настройка репозитория
Шаг 1: Создание удаленного репозитория на GitHub
Перейдите на GitHub и войдите в свою учетную запись.
Нажмите кнопку "New" для создания нового репозитория.
В поле "Repository name" введите FastMediaSorter_mobile.
Выберите тип репозитория: "Private" (рекомендуется для начала).
Важно: Не инициализируйте репозиторий с файлами README, .gitignore или license, так как проект уже существует локально.
Нажмите "Create repository".
Шаг 2: Подключение локального проекта к удаленному репозиторию
Откройте терминал или командную строку в корневой директории вашего проекта (c:\GIT\FastMediaSorter_mobile).
Свяжите локальный репозиторий с удаленным, используя команду, предоставленную GitHub на следующей странице после создания репозитория. Она будет выглядеть так:
git remote add origin https://github.com/YourUsername/FastMediaSorter_mobile.git
Убедитесь, что удаленный репозиторий успешно добавлен:
git remote -v
Вы должны увидеть две строки (для fetch и push) с URL вашего репозитория.
Шаг 3: Синхронизация и отправка исходного кода
Рекомендуется переименовать основную ветку в main, если она называется master, для соответствия современным стандартам:
git branch -M main
Отправьте существующие локальные коммиты на GitHub:
git push -u origin main
Эта команда загрузит ваши файлы и установит origin/main как вышестоящую ветку для вашей локальной ветки main.
Шаг 4: Настройка файла .gitignore
Проект уже содержит файл .gitignore. Необходимо убедиться, что он включает все стандартные для Android-разработки на Kotlin исключения, чтобы предотвратить попадание в репозиторий сгенерированных системой, локальных и временных файлов.

Шаг 5: Защита основной ветки (Branch Protection)
На странице вашего репозитория на GitHub перейдите в "Settings" > "Branches".
Нажмите "Add branch protection rule".
В поле "Branch name pattern" введите main.
Установите следующие флаги для защиты ветки:
Require a pull request before merging: Все изменения должны проходить через Pull Request.
Require status checks to pass before merging: Обязательная проверка сборки и тестов перед слиянием (настраивается после настройки CI/CD).
Require conversation resolution before merging: Все комментарии в Pull Request должны быть разрешены.
Нажмите "Create". Это установит базовые правила для качественного ведения разработки.
