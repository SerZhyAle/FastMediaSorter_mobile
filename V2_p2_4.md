# 2.4. Настройка CI/CD Pipeline

Эта инструкция описывает процесс создания CI/CD pipeline с использованием GitHub Actions для автоматической сборки, тестирования и публикации проекта.

## Основные цели
-   **Continuous Integration (CI):** Автоматический запуск проверок качества кода, unit-тестов и создание debug-сборки при каждом push в ветку `main` и при создании Pull Request.
-   **Continuous Deployment (CD):** Автоматическое создание подписанной release-версии приложения (AAB) и её публикация в GitHub Releases при создании нового тега (например, `v2.0.1`).

## Шаг 1: Создание файла Workflow
1.  В корневой директории проекта создайте папку `.github`, а внутри нее — папку `workflows`.
2.  В папке `.github/workflows` создайте новый файл с именем `android_ci_cd.yml`.

## Шаг 2: Настройка CI (Непрерывная интеграция)

Вставьте следующий код в `android_ci_cd.yml`. Этот код определяет первую часть pipeline — `ci-job`.

```yaml
name: Android CI/CD

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  ci-job:
    name: Run Checks and Tests
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Set up JDK 17
        uses: actions/setup-java@v3
        with:
          java-version: '17'
          distribution: 'temurin'

      - name: Grant execute permission for gradlew
        run: chmod +x gradlew

      - name: Setup Gradle Cache
        uses: actions/cache@v3
        with:
          path: |
            ~/.gradle/caches
            ~/.gradle/wrapper
          key: ${{ runner.os }}-gradle-${{ hashFiles('**/*.gradle*', '**/gradle-wrapper.properties') }}
          restore-keys: |
            ${{ runner.os }}-gradle-

      - name: Run lint checks
        run: ./gradlew lintDebug

      - name: Run unit tests
        run: ./gradlew testDebugUnitTest

      - name: Build debug APK
        run: ./gradlew assembleDebug

      - name: Upload debug APK
        uses: actions/upload-artifact@v3
        with:
          name: debug-apk
          path: app/build/outputs/apk/debug/app-debug.apk
```

**Описание:**
-   **`on`**: Запускает pipeline при push или pull request в ветку `main`.
-   **`ci-job`**: Название задачи.
-   **`runs-on`**: Указывает, что задача будет выполняться на виртуальной машине с Ubuntu.
-   **`steps`**: Последовательность шагов:
    1.  `checkout`: Клонирование кода репозитория.
    2.  `setup-java`: Установка JDK 17.
    3.  `chmod +x gradlew`: Предоставление прав на выполнение для Gradle Wrapper.
    4.  `cache`: Кэширование зависимостей Gradle для ускорения последующих сборок.
    5.  `lintDebug`: Запуск статического анализатора кода.
    6.  `testDebugUnitTest`: Запуск unit-тестов.
    7.  `assembleDebug`: Сборка debug-версии APK.
    8.  `upload-artifact`: Выгрузка собранного APK в артефакты сборки для возможности скачивания.

## Шаг 3: Настройка CD (Непрерывное развертывание)

Этот шаг требует безопасного хранения ключа для подписи release-сборок.

### 3.1. Подготовка ключа и секретов
1.  Если у вас нет ключа подписи, сгенерируйте его с помощью `keytool`.
2.  Конвертируйте ваш файл keystore (`.jks`) в строку формата Base64. Выполните в терминале:
    ```bash
    openssl base64 -A -in your_keystore_file.jks
    ```
3.  Скопируйте полученную строку.
4.  В вашем репозитории на GitHub перейдите в **Settings > Secrets and variables > Actions** и нажмите **"New repository secret"** для каждого из следующих секретов:
    -   `KEYSTORE_BASE64`: Вставьте сюда строку Base64 из предыдущего шага.
    -   `KEYSTORE_PASSWORD`: Пароль от вашего хранилища ключей.
    -   `KEY_ALIAS`: Alias вашего ключа.
    -   `KEY_PASSWORD`: Пароль от вашего ключа.

### 3.2. Настройка `build.gradle.kts` для подписи

Откройте файл `app/build.gradle.kts` и добавьте в блок `android { ... }` конфигурацию для подписи. Это позволит Gradle использовать секреты, переданные из GitHub Actions.

```kotlin
android {
    // ... другие настройки

    signingConfigs {
        create("release") {
            keyAlias = System.getenv("KEY_ALIAS")
            keyPassword = System.getenv("KEY_PASSWORD")
            storeFile = System.getenv("KEYSTORE_PATH")?.let { File(it) }
            storePassword = System.getenv("KEYSTORE_PASSWORD")
        }
    }

    buildTypes {
        getByName("release") {
            isMinifyEnabled = true
            proguardFiles(getDefaultProguardFile("proguard-android-optimize.txt"), "proguard-rules.pro")
            signingConfig = signingConfigs.getByName("release")
        }
    }
}
```

### 3.3. Добавление задачи CD в Workflow

Добавьте вторую задачу (`cd-job`) в конец файла `android_ci_cd.yml`.

```yaml
  cd-job:
    name: Build Release and Publish
    needs: ci-job # Запускать только после успешного завершения ci-job
    if: startsWith(github.ref, 'refs/tags/') # Запускать только при создании тега
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Set up JDK 17
        uses: actions/setup-java@v3
        with:
          java-version: '17'
          distribution: 'temurin'

      - name: Grant execute permission for gradlew
        run: chmod +x gradlew

      - name: Decode Keystore
        env:
          KEYSTORE_BASE64: ${{ secrets.KEYSTORE_BASE64 }}
        run: |
          echo $KEYSTORE_BASE64 | base64 --decode > release.jks

      - name: Build Release AAB
        env:
          KEY_ALIAS: ${{ secrets.KEY_ALIAS }}
          KEY_PASSWORD: ${{ secrets.KEY_PASSWORD }}
          KEYSTORE_PATH: ${{ github.workspace }}/release.jks
          KEYSTORE_PASSWORD: ${{ secrets.KEYSTORE_PASSWORD }}
        run: ./gradlew bundleRelease

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          files: app/build/outputs/bundle/release/app-release.aab
          tag_name: ${{ github.ref_name }}
          body: "Release ${{ github.ref_name }}"
```

**Описание:**
-   **`needs: ci-job`**: Задача `cd-job` не начнется, пока `ci-job` не завершится успешно (если они запускаются одновременно).
-   **`if: startsWith(github.ref, 'refs/tags/')`**: Условие, которое запускает эту задачу только тогда, когда событием является создание нового тега.
-   **`Decode Keystore`**: Шаг декодирует строку Base64 из секрета GitHub и воссоздает из нее файл `release.jks`.
-   **`Build Release AAB`**: Шаг выполняет сборку release-бандла (`.aab`). Переменные окружения (`env`) передают секреты в `build.gradle.kts` для подписи.
-   **`Create GitHub Release`**: Шаг автоматически создает новый релиз на GitHub, прикрепляя к нему собранный `.aab` файл.

## Шаг 4: Запуск Pipeline
1.  Сохраните и отправьте файл `android_ci_cd.yml` в ваш репозиторий.
    ```bash
    git add .github/workflows/android_ci_cd.yml
    git commit -m "feat: Add CI/CD workflow"
    git push
    ```
2.  Перейдите на вкладку **"Actions"** в вашем репозитории на GitHub, чтобы увидеть выполнение `ci-job`.
3.  Для проверки CD, создайте и отправьте новый тег:
    ```bash
    git tag -a v2.0.0-alpha1 -m "Initial release"
    git push origin v2.0.0-alpha1
    ```
4.  После этого на вкладке **"Actions"** запустится `cd-job`, и по его завершении в разделе **"Releases"** вашего репозитория появится новый релиз с прикрепленным AAB-файлом.
