# 2.3. Инициализация структуры проекта

Этот документ описывает рекомендуемую структуру пакетов и модулей для проекта `FastMediaSorter_mobile` v2. Структура основана на принципах чистой архитектуры (Clean Architecture) и паттерна MVVM, что обеспечивает разделение ответственности, тестируемость и масштабируемость кода.

## 1. Общая структура модуля `app`

Основная логика приложения будет находиться в главном модуле `:app`. Внутри `app/src/main/java/com/sza/fastmediasorter` предлагается следующая организация пакетов:

```
com.sza.fastmediasorter
├── core                # Общие компоненты и утилиты для всего приложения
│   ├── di              # Модули Hilt для внедрения зависимостей
│   ├── model           # Общие модели данных, используемые в разных слоях
│   ├── ui              # Базовые классы для UI (BaseActivity, BaseFragment, View-расширения)
│   └── utils           # Утилитарные классы (работа с файлами, сетью, датами)
│
├── data                # Слой данных: работа с источниками данных
│   ├── local           # Локальные источники
│   │   └── db            # Компоненты Room: Database, DAO, Entities
│   ├── remote          # Удаленные источники
│   │   ├── smb           # Клиент для работы с SMB
│   │   ├── sftp          # Клиент для работы с SFTP
│   │   └── cloud         # API для облачных хранилищ (Google Drive, etc.)
│   └── repository      # Реализации интерфейсов репозиториев из domain-слоя
│
├── domain              # Слой бизнес-логики (чистый Kotlin, без зависимостей от Android)
│   ├── model           # Бизнес-модели (POKO - Plain Old Kotlin Objects)
│   ├── repository      # Интерфейсы репозиториев (контракты для слоя данных)
│   └── usecase         # Use-кейсы, инкапсулирующие конкретные бизнес-задачи
│
└── ui                  # Слой представления (Presentation Layer), организованный по фичам
    ├── main            # Главный экран (список ресурсов)
    ├── browse          # Экран просмотра и сортировки медиафайлов
    ├── player          # Экран проигрывателя
    ├── settings        # Экран настроек
    └── resource        # Экраны добавления и редактирования ресурсов
```

### Описание слоев

-   **`core`**: Содержит код, который может быть переиспользован в любом другом слое или фиче. Сюда входят настройка Dependency Injection, базовые классы для UI, модели данных, которые не изменяются (например, константы), и различные хелперы.

-   **`data`**: Отвечает за предоставление данных приложению. Он содержит реализации репозиториев, код для работы с базой данных Room, сетевыми протоколами (SMB, SFTP) и облачными API. Этот слой преобразует данные из сырого формата (DTO, DB-entity) в доменные модели.

-   **`domain`**: Содержит основную бизнес-логику приложения. Этот слой не зависит от деталей реализации UI или источников данных. Он определяет, *что* приложение делает. `UseCases` в этом слое оркеструют поток данных из репозиториев для достижения бизнес-целей.

-   **`ui`**: Отвечает за отображение данных на экране и обработку взаимодействия с пользователем. Каждая фича (например, `main`, `browse`) имеет свой пакет, содержащий `Activity`/`Fragment`, `ViewModel`, `Adapter` и другие UI-специфичные классы. `ViewModel` запрашивает данные у `UseCases` и подготавливает их для отображения.

## 2. Структура ресурсов (`res`)

Ресурсы в `app/src/main/res` также рекомендуется организовывать с префиксами, соответствующими фиче, для удобства навигации:

-   **`layout`**: `activity_main.xml`, `fragment_browse.xml`, `item_resource.xml`
-   **`drawable`**: `ic_browse_filter.xml`, `bg_player_button.xml`
-   **`menu`**: `menu_browse.xml`
-   **`navigation`**: Графы навигации для Navigation Component, например `nav_main.xml`.

## 3. Структура тестов

-   **`app/src/test`**: Для локальных Unit-тестов. Структура пакетов должна повторять основную структуру (`data`, `domain`, `ui`) для тестирования `ViewModels`, `UseCases` и `Repositories`.
-   **`app/src/androidTest`**: Для инструментальных тестов, требующих Android-контекста (тестирование UI, базы данных Room).

## Порядок действий

1.  В Android Studio в режиме `Project` перейдите в папку `app/src/main/java/com/sza/fastmediasorter`.
2.  Создайте базовые пакеты: `core`, `data`, `domain`, `ui`.
3.  Внутри каждого пакета создайте подпакеты согласно предложенной структуре.
4.  Начните рефакторинг существующего кода, перемещая классы в соответствующие пакеты. Например, классы, работающие с базой данных, — в `data.local.db`, `Activity`/`Fragment` — в соответствующие пакеты внутри `ui`.
5.  Создайте базовые классы (например, `BaseActivity`, `BaseViewModel`) в пакете `core.ui` для переиспользования общего кода.
